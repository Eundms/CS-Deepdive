# eden에서 객체는 어떻게 생성되는가?
객체는 heap에서도 eden에 생성된다
힙은 스레드들이 공유하며 접근하므로 충돌 가능성이 존재한다 
### Q. 병목은 없게 이 문제를 해결하려면?  
=> eden 영역의 일부를 각 스레드들이 독립적으로 사용할 수 있게 TLAB(thread local allocation buffer)를 할당해서 객체가 생성될 수 있도록 하자 TLAB 할당 받을 때는 경합이 있을 수 있음
=> bump-the-pointer : 객체가 다음에 생성될 위치를 포인터로 가리킨다
=> TLAB 안에서 객체가 생성되는 경우를 fast-path라고 부른다 (synchronization 이슈가 없어서 빠르게 객체 생성이 가능하다)

### Q. TLAB의 남은 공간보다 더 큰 객체를 할당해야 한다면?
1. TLAB에 남은 공간이 여유가 있지만 새로 생성할 객체가 이보다 큰 경우 TLAB 밖의 eden 영역에 직접 할당한다 
2. 여유가 없으면 기존 TLAB를 retire하고 eden영역에서 새로운 TLAB를 refill한다 
=> 1,2 모두 eden의 공유 영역에서 생성되며 동기화와 JVM 로직이 개입해서 속도가 느리다 이를 slow-path라고 한다 

#### 남은 공간이 여유 있는지 판단하는 방법 : refill_waste_limit보다 크면 아직 여유가 있다고 판단
- [초기값] refill_waste_limit = TLAB size / TLABRefillWasteFraction  (-XX:TLABRefillWasteFraction=N(64))
- [1번 발생할 때 마다] refill_waste_limit += TLABwasteIncrement(-XX:TLABWasteIncrement=M(default 4, heap word 단위))

- XX:ResizeTLAB가 기본적으로 켜져 잇어서 TLAB의 사이즈가 다이나믹하게 재조정되도록 만들 수 있음
- young GC 후 각 스레드 별로 앞으로의 TLAB 크기가 재조정되고 이를 desired_size라고 부른다 
- TLAB 사이즈 조정에 영향을 미치는 주요 요소들
  - eden 사이즈
  - thread 수
  - -XX:TLABWasteTargetPercent=N (기본1, 단위x) <= 값이 클수록 desired_size가 커짐 
  - allocation rate(각 스레드 별로 eden의 얼마를 할당받았는지)와 -XX:TLABAllocationWeight=M(default 35, 단위 %) <= 값이 클수록 직전 allocation rate를 더 많이 반영  

# TLAB 관련 주요 Vm Option
- -XX:UseTLAB : TLAB 사용 여부 (기본 true)
- refill: -XX:TLABRefillWasteFraction=N(64), -XX:TLABWasteIncrement=M(heap word, 4)
- resize: -XX:ResizeTLAB(true) -XX:`TLABWasteTargetPercent`=N(%,1) -XX:TLABAllocationWeight=M(%,35)
- TLAB size
- -XX:TLABSize=N : 스레드 최초록 TLAB를 할당받을 떄 TLAB 사이즈. 단위는 bytes이며 default값은 0
- -XX:MinTLABSize=M : TLAB 크기의 최소값. 단위는 bytes이며 default값은 2048
- 대부분 그대로 사용


# GC 성능 관련 주요 지표 
- Throughput : 전체 실행 시간 중 GC에 소비된 시간은 제외하고 애플리케이션이 실행된 시간 (객체 할당은 애플리케이션 실행 시간에 포함)
- Latency : GC로 인해 애플리케이션이 멈추는 stop-the-world 시간
- Footprint : 애플리케이션이 실제로 점유하고 있는 물리적 메모리양 (물리 메모리가 적거나 많은 프로세스가 물리 메모리를 공유할 때 중요 )
- Promptness : Garbage가 된 후 해당 메모리가 다시 사용이 가능할 때까지의 시간 (RMI를 포함한 분산 환경에 중요)
- GC의 주요 측정 지표는 Throughput과 Latency
     - Throughput과 Latency는 trade-off 관계
     - generation 사이즈를 결정하는 것은 Throughput과 Latency 사이에서 절충점을 선택하는 것과 같다
